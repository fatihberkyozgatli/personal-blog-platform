# Pages from the Red Diary: Testing & Stage 5/6 Hardening

> Design spec. Agreed 2026-06-18. Builds a four-layer test suite (unit, integration, DB/RLS, E2E),
> fixes the known Stage 5/6 correctness bugs, and verifies the project docs ŌĆö then runs the
> per-stage three-way review gate and reports. Delivered as ordered phases on one feature branch.

---

## 1. Goal & Execution Model

**Goal:** raise confidence before deploy ŌĆö real test coverage at every layer, the known
Stage 5/6 bugs fixed, and the documentation verified accurate.

**Execution model (agreed):**
- The implementation writes ALL code/scripts and runs everything runnable locally: unit tests,
  integration tests (mocked Supabase), `typecheck`, `lint`, `build`.
- The **owner runs** the parts that need cloud/runtime access (no local Docker stack): the DB/RLS
  SQL (`supabase db query --linked`), the E2E suite (`npm run e2e`), and live Stage 5/6
  verification. These are written to be runnable but are not executed by the implementation.
- **E2E scope now:** anon/public + login-UI flows only. Authenticated reader/admin flows are
  deferred until dedicated test accounts exist.
- **Stage 5/6 scope:** fix the known correctness bugs and cover them with tests; the owner does the
  live owner test.

**Branch & flow:** one feature branch `feat/testing-and-hardening`; commits split per phase (each
phase is its own commit set, never one mega-commit); per-task review during subagent execution; the
three-way review gate (`/code-review`, `/security-review`,
`/ui-ux-pro-max:ui-ux-pro-max`) runs ONCE at the end over the whole branch diff; then a combined
report; merge to `main` only on the owner's explicit command.

## 2. Global Constraints

- TypeScript strict; no code comments; no emojis; no `Co-Authored-By` trailer.
- Validate inputs with Zod in `lib/validations/` (this initiative finishes centralizing that).
- Tests must assert real behavior, not mocks-asserting-mocks.
- Vitest has no `.env.local`; modules importing `@/lib/supabase/server` throw at import unless the
  server client is lazily imported OR the module is `vi.mock`ed (see Phase 2). The `@/` vitest
  alias is path-scoped (`^@/`) ŌĆö keep it that way.
- Security lives in the DB (RLS); the RLS layer (Phase 3) is the most load-bearing test layer.
- Brand name: "Pages from the Red Diary" (centralized in `lib/site.ts`).

## 4. Phase 1 ŌĆö Unit Tests (+ schema extraction)

The requested `contact/newsletter/post/category` unit tests presuppose importable Zod schemas.
Today contact/newsletter/post validate with **inline** schemas and category has **no** Zod. So:

**Extract / create schemas in `lib/validations/` and wire the actions to import them:**
- `contact.ts` ŌĆö move the inline schema out of `lib/actions/contact.ts` (export `contactSchema`).
- `newsletter.ts` ŌĆö move the inline schema out of `lib/actions/newsletter.ts` (export `newsletterSchema`).
- `post.ts` ŌĆö move the inline `postSchema` out of `lib/actions/admin.ts` (export `postSchema`).
  Treat `content` as a Tiptap **doc object**, not a string: validate it is an object with
  `type === "doc"` and a `content` array (e.g. `z.object({ type: z.literal("doc"), content:
  z.array(z.unknown()) }).passthrough()`) ŌĆö NO strict ProseMirror node validation (it breaks when
  extensions change) and NO string `min-length`. `savePost` is updated to `JSON.parse` the form's
  content string FIRST, then validate the assembled object with `postSchema` (equivalent end
  behavior; the "empty body" guard can stay in the action).
- `category.ts` ŌĆö NEW `categorySchema` validating the actual `createCategory` INPUTS: `name`
  required, `icon` optional. `slug` is generated by the action (`slugify(name)`), not user input, so
  it is NOT in the schema; `description` is not collected in v1. Refactor `createCategory` in
  `lib/actions/admin.ts` to validate with `categorySchema`.

**Then add unit tests:**
- `lib/validations/contact.test.ts` ŌĆö rejects invalid email, requires name + body, optional subject.
- `lib/validations/newsletter.test.ts` ŌĆö rejects invalid email, accepts valid.
- `lib/validations/post.test.ts` ŌĆö requires title; rejects a bad `status`; accepts a valid Tiptap
  doc object for `content` and rejects a non-doc (`{}`, a string, an array). No assertions about
  internal node structure.
- `lib/validations/category.test.ts` ŌĆö requires name, optional icon.
- `lib/utils/format.test.ts` ŌĆö `formatDate` output and the reading-time/format helpers are correct.
- `lib/tiptap/render.test.ts` ŌĆö `renderPostHtml` strips unsafe HTML (e.g. `<script>`, `onerror`,
  `javascript:` href), keeps allowed tags, and returns `""` for a null/empty doc.

(`about` fallback is already covered by `lib/data/about.test.ts`.)

**Done when:** `npm test` is green with the new files; each action still works via its extracted
schema (typecheck + the existing flows unaffected).

## 5. Phase 2 ŌĆö Integration Tests (mocked Supabase)

Server Action + validation behavior, with the Supabase client and auth mocked. Use `vi.mock` for
`@/lib/supabase/server`, `@/lib/auth`, and `next/cache` (mocking these also avoids the `lib/env.ts`
import-throw). Build a REUSABLE query-builder mock helper at `tests/helpers/mock-supabase.ts` (a
chainable `from().select().eq()ŌĆ”` / `insert` / `upsert` stub that records calls and returns a
configurable `{ data, error }`) plus a configurable `getCurrentUser` mock ŌĆö without it the chain
mocking degrades into "mocks testing mocks". Integration specs live in `tests/integration/*.test.ts`,
importing the helper relatively and the actions via `@/lib/actions/...`.

**Targets:**
- `signIn` with invalid data returns an error (no redirect).
- `safeNext` / redirect helper returns a safe path for hostile input.
- `updateAbout` rejects an invalid payload shape (and rejects a non-admin caller).
- newsletter `subscribe` treats a duplicate email (Postgres `23505`) as success.
- contact `sendMessage` inserts a valid message (mock insert called with parsed data).
- admin actions reject a non-admin caller before any write.

**Done when:** these tests pass under `npm test` and genuinely exercise the action logic (the mock
records calls; assertions check the action's branching, not the mock itself).

Extend `supabase/tests/rls_checks.sql` in its existing `begin; set local role ŌĆ”; do $$ ŌĆ” assert ŌĆ”;

- an admin CAN UPDATE `site_settings`.
- `media` writes (insert/update/delete on `storage.objects` for the `media` bucket) succeed only

  reruns never collide.

(Existing checks already cover: anon cannot read `posts.content`; anon/reader cannot see

**Owner runs:** `supabase db query --linked --file supabase/tests/rls_checks.sql` (or via the

**Done when:** the script is complete and self-asserting (raises on any violation); the owner's run

## 7. Phase 4 ŌĆö Stage 5/6 Bug Fixes (+ tests)

Fix the known `TODOs.md` correctness bugs and cover what is testable:

- **Tag filter P0** (`lib/data/posts.ts` `getPosts`): resolve `tagSlug` ŌåÆ `tag_id`, fetch matching

**Tests:** unit-test the tag-resolution and any pure helpers; integration-test `toggleLike` with the

## 9. Phase 6 ŌĆö Scripts, Review Gate, Report

**`package.json` scripts:**
```
"test": "vitest run",
"test:watch": "vitest",
"test:rls": "supabase db query --linked --file supabase/tests/rls_checks.sql",
"e2e": "playwright test",
"e2e:ui": "playwright test --ui",
"verify": "npm run typecheck && npm run lint && npm test && npm run build",
"verify:full": "npm run verify && npm run test:rls && npm run e2e"
```

`verify` is local + deterministic (typecheck, lint, unit/integration tests, build) ŌĆö safe for anyone
or CI. `verify:full` ALSO runs `test:rls` (needs a linked Supabase) and `e2e` (needs Playwright
browsers + a running app + seeded DB), so it WILL fail without that setup ŌĆö that is expected, not
"broken". Document this distinction in `README.md`.

**Review gate:** run the three reviews in parallel over the whole-branch diff
(`/code-review`, `/security-review`, `/ui-ux-pro-max:ui-ux-pro-max`), present one combined compared
report (overlap vs unique, by severity), and WAIT for the owner's command before any fixes or merge.

**Report:** a final combined report of what was built, what passed locally, and exactly what the
owner still needs to run (RLS, E2E) and configure (the deploy prerequisites).

**Done when:** scripts are in; the gate report is delivered; the owner decides on findings + merge.
